Bienvenidos al proyecto ADISAN

El proyecto de referencia en la gestión sanitaria educativa mediante simulación

¿Qué es ADISAN?

ADISAN es un proyecto educativo multidisciplinar para el aprendizaje de alumnos de Formación Profesional mediante simulación.
ADISAN ha nacido para mejorar el aprendizaje de los alumnos en dos familias de Formación Profesional:
Familia sanitaria

ADISAN es un simulador de gestión administrativa sanitaria. Los profesores de Ciclos Formativos de la familia Sanitaria lo utilizan para qu elos alumnos realizen práctias académicas simulando el entorno laboral.

Los profesores que usamos ADISAN para simular el entorno sanitario laboral estamos desarrollando y compartiendo materiales didácticos para explotar el simulador de gestión administrativa sanitaria.

Familia informática

ADISAN se ha desarrollado para que los alumnos del Ciclo Formativo de Grado Superior en Desarrollo de Aplicciones Web (DAW) tengan un modelo de referencia para aprender a desarrollar una aplicación web compleja.

La estructura de ADISAN está pensada para que los alumnos del ciclo formativo DAW la utilizen para aprender modificando su código o copiandolo en sus proyectos.

¿Por qué se desarrolla ADISAN?

Los sistemas de información sanitaria en producción, en general, son poco accesibles por parte de actores externos. Este hecho es debido principalmente a la sensibilidad de los datos que manejan y al gran volumen que gestionan los mismos. Además son sistemas complejos y con costes elevados de desarrollo y mantenimiento. Por todo ello, los alumnos del ciclo formativo de grado superior denominado Técnico superior en documentación y administración sanitaria (DAS) actualmente no disponían de un sistema de información sanitario que fuera operativo, próximo a la realidad de la gestión sanitaria en nuestro país y que permita explicar, aplicar, practicar y desarrollar los conocimientos que van adquiriendo en dicho ciclo formativo.

Los alumnos del Ciclo formativo de grado superior denominado Desarrollo de aplicaciones web (DAW) tienen que capacitarse en desarrollar aplicaciones web, pero no disponían de un sistema complejo en producción que les sirva de referente para practicar y para desarrollar sus proyectos. La mera suma de actividades o ejercicios de programación a lo largo del curso no es suficiente para que acaben desarrollando una aplicación web completa por ellos mismos. Necesitaban un sistema de referencia para ver cómo interactúan las distintas tecnologías que se necesitan para hacer funcionar una aplicación web.

Para suplir estas carencias del entorno educativo se decidió desarrollar un sistema que se pudiera usar conjuntamente en sanidad y desarrollo web.

¿Qué pretendemos con este proyecto?

Para alumnos de la familia Sanitaria (ciclo Documentación y administración sanitaria)
  • Modelizar procesos de atención sanitaria para que los alumnos los entiendan y los apliquen.
  • Integrar distintas áreas de la gestión sanitaria en un mismo sistema.
  • Simular la gestión de los centros de atención sanitaria.
  • Preparar materiales didácticos para apoyar el uso del programa.
Para alumnos de la familia Informática, (ciclo Desarrollo de aplicaciones web)
  • Aprender a estructurar el desarrollo de un proyecto web.
  • Simular la creación de un proyecto complejo en un entorno de trabajo.
  • Disponer de una aplicación compleja que el alumno pueda entender, modificar y copiar para aprender a crear sus propios proyectos.
  • Trabajar en grupo para desarrollar partes de la aplicación con sistemas de control de versiones.
En definitiva, el objetivo es preparar mejor a los alumnos para que tengan una mejor integración profesional.
¿Qué tecnología se ha usado para desarrollar proyecto?
En la parte cliente

La versión actual está desarrollada con AngularJS. La aplicación se ha desarrollado emulando componentes genéricos con plantillas y scopes. Las datos se obtinen del servidor en formato JSON mediante peticiones AJAX. Los datos que vienen del servidor incorporan metadatos para facilitar la reutilización de componentes. Las sesiones se mantienen en el lado del servidor. En cada petición se consulta al servidor el estado de la sesión, que informa del usuario logueado y su perfil. Se ha usado el enrutamiento estándar de AngularJS para cliente, donde cada ruta se chequea en función del estado de la sesión informado por el servidor. En los formularios las validaciones se realizan también de forma estándar.

En la parte servidora

Se ha utilizado un servlet que actúa de controlador para recibir las peticiones y servirlas mediante JSON encaminándolas hacia el nivel inferior de servicios (service). Los servicios normalmente son generados mediante factorías, que emulan la inyección de dependencias sin frameworks adicionales. El nivel service recibe las peticiones, crea la conexión con la base de datos y utiliza uno o varios objetos del nivel inferior de acceso a datos (DAO) para servirlas. Cuando recibe el resultado de la petición lo devuelve en formato JSON al controlador. El nivel DAO además implenta la gestión de perfiles y sus validaciones de seguridad, así como el acceso a la base de datos específico para cada tipo de petición. Ambos sercicios (service y dao) ofrecen un conjunto de implementaciones genéricas mediante Java Reflection para ofrecer servicios comunes que se repiten en la aplicación. La conexión se genera mediante su correspondiente factory desde el nivel servicio y fluye hasta servir la petición. Existen distintas implementaciones para usar cualquiera de los pools de conexiones más conocidos en Java. Los metadatos se guardan en los beans en forma de anotaciones. La base de datos que se ha usado es MySQL. También se ha usado google GSON para las conversiones JSON a bean y maven como gestor de dependencias.

¿Cómo se tratan los datos personales en la aplicación?
  • ADISAN no almacena ningún dato de tipo personal.
  • Los alumnos, profesores y usuarios en general que utilizan esta aplicación acceden mediante un nombre de usuario, que es un texto no asociable desde la aplicación a una identidad física, ya que ésta no guarda ningún otro dato asociado a dicho nombre de usuario.
  • Los datos médicos y sanitarios de la aplicación son todos ficticios. Las identidades de pacientes y personal sanitario, son generadas aleatoriamente. La información concerniente a los historiales médicos de los pacientes también es ficticia.
  • Los usuarios no tienen permiso para almacenar ninguna información ajena a los fines didácticos con que ha sido concebido el proyecto. Si un usuario incumple este término, su cuenta será inmediantamente baneada y la información eliminada.
  • Los alumnos deben acceder a la aplicación únicamente bajo la supervisión del profesor que los gestiona en el sistema.
